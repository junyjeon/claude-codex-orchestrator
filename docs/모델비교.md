# Claude vs Codex 모델 비교

이 문서가 모델 버전과 벤치마크의 **단일 소스**다. 모델이 업그레이드되면 이 파일만 수정한다. 다른 문서(README, CLAUDE.md, 설계.md)는 이 파일을 참조만 한다.

현재 비교: Opus 4.6 vs Codex 5.4 (2026-03-06 기준)


## 벤치마크

| 벤치마크 | Opus 4.6 | Codex 5.4 | Codex 5.3 | 측정 대상 |
|----------|----------|-----------|-----------|-----------|
| SWE-Bench Verified (vals.ai) | 79.2% | 77.2% | 75.2% | 실제 버그 수정 |
| SWE-Bench Pro Public | 80.8%* | 57.7% | 56.8% | 실제 버그 수정 |
| Terminal-Bench 2.0 | 65.4% | 미제출 | 77.3% | 터미널 에이전틱 코딩 |
| GPQA Diamond | 77.3% | - | - | 학술 추론 |
| MMLU Pro | 85.1% | - | - | 전문 지식 |
| Aider Polyglot | ~72% | 88.0% | 86.7% | 다국어 코드 생성 |
| LFG Benchmark | 9.25/10 | - | 7.5/10 | 복잡한 기능 구현 |
| OSWorld-Verified | - | 75.0% | 74.0% | 컴퓨터 사용 |
| GDPval | - | 83.0% | 70.9% | 지식 업무 |
| Toolathlon | - | 54.6% | 51.9% | 도구 사용 |

*SWE-Bench 주의: Anthropic은 Verified, OpenAI는 Pro Public 버전을 리포팅한다. vals.ai가 동일 조건 비교를 제공한다. Terminal-Bench 2.0은 GPT-5.4 출시 직후라 아직 미제출.*


## 5.3 → 5.4 변화

코딩 성능은 소폭 개선이다. SWE-Bench Pro에서 56.8% → 57.7%로 0.9%p, Aider Polyglot에서 86.7% → 88.0%로 1.3%p 올랐다. 코딩 자체보다는 효율성과 에이전틱 능력에 집중한 업그레이드다.

에이전틱 능력이 대폭 강화됐다. GDPval(지식 업무) 70.9% → 83.0%로 12.1%p 뛰었고, OSWorld(컴퓨터 사용)는 75.0%로 인간(72.4%)을 넘었다. "코딩 전문가"에서 "에이전틱 만능 모델"로 전환한 것이 5.4의 핵심 방향이다.

토큰 효율이 대폭 개선됐다. 일부 작업에서 토큰 사용이 47% 줄었다. 같은 작업을 더 빠르고 싸게 처리한다.

컨텍스트 윈도우가 1M 토큰으로 늘었다. 긴 코드베이스를 한 번에 다룰 수 있다.

5.3-Codex의 코딩 강점에 도구 사용(tool search)과 컴퓨터 사용(computer use)을 하나의 모델로 통합했다. 5.3은 코딩(Codex)과 범용을 분리했지만 5.4는 통합 모델이다.


## 핵심 특성 차이

Opus는 ceiling이 높고 variance도 높다. 복잡한 작업에서 깊이 생각하고 전체를 구현한다. 하지만 간단한 작업에서 가끔 과잉 설계한다.

Codex는 ceiling은 낮지만 variance도 낮다. 결과물이 예측 가능하고 안정적이다. 더 빠르고 토큰당 비용이 낮다. 5.4부터 토큰 효율이 더 개선됐다.

한마디로: Opus는 자율 주행, Codex는 반자동 주행이다.


## 작업별 최적 모델

### Claude가 앞서는 영역

아키텍처 설계와 시스템 계획이다. 추론 깊이가 다르다. 멀티에이전트 구조 설계, 데이터베이스 스키마, API 설계에서 Codex보다 깊은 트레이드오프 분석을 한다.

복잡한 멀티파일 기능 구현이다. 여러 파일에 걸친 일관성이 필요한 작업에서 강하다.

버그 수정과 디버깅이다. SWE-Bench에서 일관되게 앞선다. 기존 코드를 읽고 문제를 진단하고 최소한의 변경으로 수정하는 능력이 압도적이다.

보안 감사와 코드 리뷰다. 추론 깊이가 곧 보안 분석 깊이다.

한국어 소통이다. 한국어 지시 이해와 문서 생성에서 더 자연스럽다.

### Codex가 앞서는 영역

터미널 기반 코딩과 스크립팅이다. CLI 도구, 셸 스크립트, 터미널 자동화에서 더 정확하다.

단일 함수/모듈 코드 생성이다. 명확하게 정의된 함수 하나를 빠르게 생성할 때 효율적이다.

반복적/정형화된 구현이다. CRUD 엔드포인트, 미들웨어, 유틸리티 함수처럼 패턴이 명확한 작업에서 낮은 variance가 장점이다.

보일러플레이트 생성이다. 설정 파일, 테스트 스캐폴딩, 타입 정의 같은 반복 작업이다.

에이전틱 도구 사용이다 (5.4 신규). tool search, computer use 등 도구 연쇄 작업에서 5.4가 강화됐다.


## 위임 판별 기준

"프로젝트 컨벤션 없이도 그대로 쓸 수 있는 코드인가?"

Yes → 위임 가능 (자족적 작업):
- 프로젝트와 무관한 bash/shell 스크립트
- 범용 유틸리티 (sort, hash, parse, format)
- 설정 파일, scaffold 생성
- 1회성 자동화 스크립트

No → 직접 처리:
- 프로젝트 컨벤션이 적용되는 모든 코드
- 기존 코드와 통합되는 기능
- 비즈니스 로직, 상태 관리, UI 컴포넌트
- 버그 수정, 리팩토링, 보안 관련


## 맥락 단절 비용

Codex에 위임하면 CLAUDE.md, rules/, 세션 히스토리가 전달되지 않는다. codex_generate의 context 필드에 Claude가 직접 맥락을 넣어야 하지만, 압축 과정에서 정보가 유실된다. 위임 범위가 넓을수록 품질 저하가 커진다.

codex_review는 예외다. Claude가 리뷰할 코드를 직접 전달하므로 맥락 손실이 없고, 다른 모델의 관점이라는 고유한 가치가 있다.


## 조합 전략

가장 가치 있는 조합은 파이프라인이 아니라 교차 리뷰다:

```
1. Claude: 설계 + 구현 + 자체 리뷰 (code-reviewer)
2. Codex: codex_review로 교차 리뷰 (다른 관점)
3. Claude: 리뷰 결과 반영
```

codex_generate는 자족적 작업에만 쓴다:

```
Claude: "이 bash 스크립트 필요해" → codex_generate → Claude: 결과 검증 후 적용
Claude: "범용 파서 함수 필요해" → codex_generate → Claude: 프로젝트 스타일로 조정
```


## 모델 업데이트 체크리스트

모델이 업그레이드되면:
1. 이 파일의 벤치마크 표 업데이트
2. "현재 비교" 라인의 모델명/날짜 수정
3. 작업별 최적 모델 섹션 재검토
4. suggest.ts의 라우팅 가중치 조정 필요 여부 확인
5. 다른 파일은 수정 불필요 (모델 버전을 참조하지 않으므로)
